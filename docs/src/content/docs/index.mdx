## What is Kubelings?

**Kubelings** is hands-on Kubernetes practice in the spirit of *rustlings* and
*golings*: a sequence of small, broken-on-purpose clusters, each with an automated
check that only passes when you've genuinely fixed the problem.

**Best for** engineers preparing for **CKA / CKAD / CKS**, or anyone who wants
Kubernetes reflexes — not just theory. Beginner → advanced, free, and it runs the
same on a browser VM or your own laptop.

## The ten-module curriculum

From first pod to control-plane internals and platform engineering, with real,
cited production incidents reproduced as runnable lessons along the way.

| # | Module | You learn | Lessons |
|---|--------|-----------|---------|
| 01 | **Foundations** | pods, Deployments, Services, the triage loop | 7 |
| 02 | **Workloads** | HPA, StatefulSets, PDBs, OOMKill, probes, VPA, KEDA | 21 |
| 03 | **Config & Storage** | ConfigMaps, Secrets, PV/PVC, Helm, Kustomize | 10 |
| 04 | **Networking** | Ingress, Gateway API, NetworkPolicy, CoreDNS, CNI | 11 |
| 05 | **Scheduling & Placement** | affinity, taints, topology spread, preemption | 5 |
| 06 | **Security** | RBAC, Pod Security, Gatekeeper, Kyverno, Falco, cosign | 18 |
| 07 | **Internals** | API server, etcd, scheduler, CRDs, operators, HA | 12 |
| 08 | **Observability & SRE** | SLO burn-rate, OTel tracing, node triage, upgrades | 10 |
| 09 | **War Stories (capstone)** | multi-concept cascades from real postmortems | 7 |
| 10 | **Platform Engineering** | Argo CD, Flux, Capsule, Cluster API, Crossplane | 6 |

[Browse every lesson, with skill outcomes and the incident write-ups →](/catalog/)

Every lesson is one of four types — three **hands-on** (you fix a live cluster,
verify-gated) and one **read-only**:

<ul class="kind-legend">
  <li><span class="kind-tag lab">lab</span> hands-on concept lesson</li>
  <li><span class="kind-tag incident">incident</span> hands-on real production incident, reproduced</li>
  <li><span class="kind-tag drill">drill</span> hands-on synthetic failure pattern</li>
  <li><span class="kind-tag read">read</span> read-only runbook, no tasks</li>
</ul>

## What makes it different

<CardGrid>
  <Card title="Interactive Go TUI" icon="laptop">
    Press `↵` on a lesson — it spins up a 3-node `kind` cluster, injects the
    broken state, and drops you into a shell. Progress markers per lesson, no YAML
    editing to start. See [The TUI](/guides/tui/).
  </Card>
  <Card title="Real incidents, reproduced" icon="random">
    40+ cited production postmortems — Zalando, Reddit, OpenAI, Monzo, Grafana —
    reproduced as runnable lessons on the same broken state the original team
    faced. Real vs synthetic is labeled honestly. Browse them in the
    [Catalog](/catalog/).
  </Card>
  <Card title="Check-driven" icon="approve-check">
    Each lesson ships `init` + `verify` tasks; the check turns green only when the
    cluster is genuinely in the target state — no self-grading.
  </Card>
  <Card title="Hardened by design" icon="approve-check-circle">
    `init`/`verify` scripts run in a confined context — they can't reach outside
    the cluster. Your host filesystem, credentials, and network are never touched.
    See [Security](/reference/security/).
  </Card>
</CardGrid>
